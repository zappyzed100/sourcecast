## 1. 目的

著作権上利用可能な歴史資料を自動収集し、面白い題材を選定して、出典付きの日本語台本・VOICEVOX音声・簡易スライド動画をほぼ無管理で生成する。

本システムは「大量転載」ではなく、複数資料から事実を抽出し、独自構成の短い歴史番組へ変換する。公開物には必ず出典と素材ライセンスを表示する。

### 1.1 成功条件

本システムの優先順位は次のとおりとする。

1. 権利不明素材と未裏付け事実を公開しない。
2. 誤りが判明した回を迅速に訂正・告知できる。
3. 出典と生成過程を第三者が追跡できる。
4. 1回あたりの管理時間を段階的に5分以内へ減らす。
5. 上記を満たした範囲で、投稿頻度と収益性を高める。

毎日投稿は成功条件に含めない。安全な候補がない日は公開しない。

### 1.2 本仕様における用語

- **候補ソース**：調査対象として登録された段階。公開利用はまだ許可されていない。
- **許可ソース**：規約・取得方法・代表資料で検証され、本番許可レジストリに登録されたもの。
- **資料**：ページ、書籍、画像、音源、地図等の個別オブジェクト。許可ソース内でも資料単位で判定する。
- **主張**：台本中の外部検証可能な事実文。
- **根拠**：主張を支える原資料内の位置が特定された箇所。
- **公開版**：特定時点の原稿・出典・クレジットをまとめた不変スナップショット。

## 2. 基本方針

- 月額API費は原則0円とする。
- LLMはOpenRouterの無料モデルを使用する。
- 情報収集は許可済みソースだけを対象とする。
- URL、取得日時、著者・提供者、ライセンス、原文抜粋を保存する。
- 出典は可能な限り恒久リンクで保存する（Wikipediaは版指定oldid付きURL、DOI、アーカイブ機関のレファレンスコード等）。
- 外部アーカイブへの保存は、対象サイトの規約とアーカイブ側の受付条件を満たす公開ページだけに任意実行する。保存失敗を公開阻害条件にせず、アーカイブURLを権利許諾の証拠として扱わない。
- 採用ソースごとに利用規約のスナップショットを保存し、定期的に再確認する。
- LLMが生成したURLを出典として採用しない。
- 権利情報が不明な文章・画像・音声は使用しない。
- 事実と、資料の文章・画像の著作権を別々に扱う（事実そのものは著作権の対象ではないが、表現・画像には権利がある）。
- 公開後も、使用資料と生成過程を追跡できるようにする。
- クローラーで取得可能であること、robots.txtで許可されていること、著作権上利用可能であること、サイト規約上再利用可能であることを、それぞれ別の判定として記録する。
- 法的評価をLLMへ委ねない。機械判定は明示ルールによる一次スクリーニングに限定し、境界事例は不採用または人手確認とする。
- 公開処理は失敗時閉鎖（fail closed）とし、不明・タイムアウト・外部API障害を許可として扱わない。

## 3. 成果物

1回の実行につき、以下を生成する。

- 5〜10分程度の読み上げ台本
- VOICEVOX（ずんだもん）による音声（BGM・効果音を含む場合は§10Eに従う）
- 16:9のスライド動画（静止画、地図、年表、字幕）
- サムネイル
- 動画タイトル、概要欄、章立て、タグ
- 出典一覧
- 素材ごとのライセンス・クレジット一覧（画像・音声・フォントを含む）
- 判定・生成ログ

初期版ではYouTube等への自動公開は行わず、「公開待ち」フォルダへ出力する。十分な実績が得られた後、条件を満たす回だけ自動公開できる構成とする。

## 4. 対象コンテンツ

### 4.1 採用する題材

- 歴史上の意外な出来事
- 人物の知られざる逸話
- 発明、制度、生活文化の起源
- 同時代の複数地域を結ぶ出来事
- 通説と一次資料の違い
- 「今日の日付」に関連する歴史

### 4.2 原則として除外する題材

- 存命人物への否定的評価
- 出典が1件しかなく、裏付けられない逸話
- 戦争犯罪、差別、性暴力等を娯楽的に扱う構成
- 現代の政治対立を直接あおる題材
- 医療・法律等について現代人の行動を誘導する内容
- 出典間で重大な矛盾が解消できない内容

## 5. 情報源

本節のサイト一覧は**候補カタログ**であり、掲載されているだけでは自動収集・転載・商用利用を許可しない。本番処理が参照する唯一の許可情報は、別途管理する機械可読な `source_registry.yaml` とする。

候補を本番許可へ昇格するには、以下をすべて満たす必要がある。

1. 公式の利用規約・ライセンスページを確認し、URLと取得時点のハッシュを保存する。
2. 正式なAPI・RSS・ダンプ・IIIF等の取得方法とレート制限を確認する。
3. 代表資料3件以上で、権利メタデータを同じ手順で取得できることをテストする。
4. 商用利用、改変、再配布、クレジット、ロゴ・第三者著作物の例外をフィールド化する。
5. 自動テスト用の許可例・拒否例を最低1件ずつ保存する。
6. 担当者の承認コミットを経て `status: approved` にする。

`status` は `candidate / approved / suspended / rejected` の4値とし、既定値は `candidate` とする。`approved` 以外は本番素材として利用しない。

### 5.1 利用区分

ソースと資料を、次の4区分で管理する。区分はソース単位で初期値を持ち、資料単位で上書きできる。

| 区分 | 意味 | 例 |
|---|---|---|
| A | 個別資料の公開利用候補。資料単位の判定通過後に限り、画像・短い引用等を成果物へ掲載できる | CC0の美術館画像、保護期間満了と確認できた書籍 |
| B | 内部調査のみ。表現・画像は公開物へ転載しない | 転載条件が厳しい資料、NCライセンスの地図 |
| C | 個別規約ソース。標準ライセンス以外の利用条件を構造化し、用途ごとに判定する | VOICEVOX、独自規約BGM |
| D | メタデータのみ。書誌・語彙・座標等に限定する | 書誌API、典拠データ |

区分Aは自動許可を意味しない。最終的な公開可否は、資料単位の `rights_decision` が `allow_public_use` であることを必要とする。

### 5.2 ライセンス正規化と自動判定

取得した権利表示を `normalized_license_id` へマッピングし、機械判定する。

| 権利表示の例 | 正規化ID | 扱い |
|---|---|---|
| CC0 | `cc0` | 公式提供元の資料単位表示であり、第三者権利例外がない場合に区分A候補 |
| Public Domain Mark | `pdm` | ライセンスではなく状態表示。提供元、対象地域、第三者権利、規約を確認後に区分A候補 |
| CC BY | `cc-by` | 採用。クレジット必須 |
| CC BY-SA | `cc-by-sa` | 採用。クレジット必須。SA（継承）は文章・画像を翻案して取り込んだ場合にのみ及ぶため、台本は事実ベースで独自作成し、SA素材の文章は引用の範囲でのみ使う |
| 政府標準利用規約（第2.0版） | `gov-jp-2.0` | CC BY 4.0互換として採用。ページ内の例外表示（第三者著作物・ロゴ等）を除外 |
| RightsStatements: No Known Copyright / No Copyright - United States | `nkc` / `noc-us` | 自動採用しない。§5Aのスクリーニング後も原則人手確認 |
| Open Government Licence（英） | `ogl` | CC BY相当として採用 |
| サイト独自規約 | `custom-<source_id>` | 区分C。規約スナップショット＋利用条件のフィールド化（商用可否・改変可否・クレジット要否・再配布可否） |
| CC BY-NC、CC BY-ND、In Copyright、表示なし・不明 | `nc` / `nd` / `inc` / `unknown` | 不採用（区分Bとしての事実調査利用のみ検討可） |

- ドメインが許可済みでも、個別資料の権利表示が不明なら不採用とする。
- 利用規約、robots.txt、アクセス制限も遵守する。「PDだが規約で商用再利用に制限があるサイト」（例：Gallica）は、著作権ではなく契約の問題として規約側を優先する。
- `normalized_license_id` だけで判定せず、`commercial_use`、`modification`、`redistribution`、`attribution`、`share_alike`、`third_party_exception`、`territory` を別フィールドで保持する。

## 5A. 日本法での機械スクリーニング基準

本節は、明白な不採用資料を除外するための一次スクリーニングであり、法的な最終判断ではない。**判定不能・境界事例は不採用**とし、年数計算は資料取得のたびに現在日付で再計算する。自動で `allow_public_use` にできるのは、公式メタデータが明示的にCC0、CC BY、CC BY-SA、政府標準利用規約、または「インターネット公開（保護期間満了）」等を示し、例外素材がない資料を基本とする。年代計算だけで許可できる資料種別は、専門家レビュー後に個別に解禁する。

本節は2026年7月時点の一般的な整理であり、実装時は文化庁、e-Gov法令、国立国会図書館等の一次情報を参照する。主要参照先は本書末尾の付録Aに固定し、確認日と内容ハッシュを保存する。

1. **原則（個人名義）**：著作権は著作者の死後70年。暦年主義により、没年＋70年の12月31日まで保護（＋71年の1月1日にPD入り）。
2. **移行基準**：日本の著作物で戦時加算等の特例がなく、著作者と没年を確認できる場合、**1967年12月31日以前に死亡した著作者の作品は保護期間満了候補**（旧50年制で満了済みの権利は2018年の延長で復活しない）。1968年以降没は死後70年を基本に計算する。翻訳、共同著作、映画、外国作品等は別規則を優先する。
3. **無名・変名・団体名義**：公表後70年。同様に、1967年以前に公表された団体名義著作物は満了済みでPD。
4. **映画**：公表後70年。ただし**1953年以前に公開された映画は保護期間満了**（2003年末満了、最高裁判例で確定）。原作・脚本・音楽は別の著作物として1へ。
5. **写真（旧法特例）**：国立国会図書館の案内に基づき、**1957年12月31日までに公表された写真**は保護期間満了候補とする。ただし、公表年、翻案・復刻物、写真に写り込んだ別著作物、所蔵サイト規約を別途確認する。自動許可の解禁までは人手確認対象とする。
6. **戦時加算**：対象国と取得時期により日数が異なる。例として米・英・豪・加・仏は3,794日、ブラジル3,816日、オランダ3,844日、ノルウェー3,846日、ベルギー3,910日、南アフリカ3,929日、ギリシャ4,180日等であり、一律3,794日では安全側にならない。国籍・著作権取得日・対象条約を確定できない外国作品は年代計算で自動許可しない。
7. **著作隣接権（音源）**：レコード、実演、楽曲、編曲、翻訳、復刻音源、提供サイト規約を別々に判定する。旧法・経過措置・戦時加算が絡むため、「1967年以前」という単独条件で自動許可しない。歴史音源はMVP対象外とし、将来も3層以上の判定ログと人手承認を必須とする。
8. **権利の目的とならないもの（著作権法13条）**：憲法その他の法令、告示・訓令・通達、判決等は自由利用できる。
9. **政府標準利用規約2.0**：CC BY 4.0互換として扱う。第三者権利部分・例外表示は除外。
10. **翻訳・翻刻・校訂**：原文がPDでも、翻訳者・校訂者に別の権利が発生し得る。翻訳文は翻訳者の没年で再判定する。台本用の現代語訳・翻訳は自前で行う。
11. **米国PD ≠ 日本PD**：`noc-us` 等の表示は米国での状態にすぎない。1〜7で日本法上の判定をやり直す。
12. **契約は著作権と別**：PDでもサイト規約が商用再利用を制限する場合（Gallica等）は規約に従う。
13. **肖像・パブリシティ・名誉・プライバシー・商標**：著作権とは別の問題として記録する。存命人物を除外するだけでは足りないため、広告的利用、遺族への影響、ロゴ、文化財所有者の表示条件等も公開ゲートで確認する。

機械スクリーニング結果は `allow_public_use / internal_research_only / manual_review / deny` の4値とする。入力不足、計算例外、規約取得失敗、国籍不明、第三者権利表示ありはすべて `manual_review` または `deny` とし、自動公開へ進めない。

## 6. 処理フロー

本システムの処理を次の7工程に固定する。

1. **歴史資料の自動収集**
   - 許可リストに登録されたWikipedia、公的機関、博物館、図書館等だけを対象にする。
   - API、RSS、データセットを優先し、必要な場合だけクローラーを使用する。
   - 本文と同時に、原典URL、権利表示、ライセンス証拠、取得日時を保存する。
2. **機械的な題材選出**
   - LLMを使用せず、日付、季節、歴史上の日付、現在のニュース見出しに含まれる語、既出度、資料数から候補を選ぶ。
   - 悪感情を呼ぶニュースとの不適切な関連付けを防ぐ安全フィルターを通す。
3. **OpenRouterによる原稿化・面白さ評価**
   - 無料モデルで事実候補抽出、題材の面白さ採点、台本作成を順に行う。
   - LLM実行前に資料の権利ゲートを通し、送信を許可された本文だけを入力する。
   - 中心的主張を原則2系統以上の独立資料で照合し、根拠位置を特定できない文章を削除する。
4. **音声・スライド生成**
   - VOICEVOXずんだもんで読み上げ音声を生成する。
   - 権利確認済み素材、自作地図、年表、字幕からスライド動画を生成する。
5. **関連書籍の機械検索**
   - LLMを使用せず、人物名、事件名、時代、ISBN、件名標目から関連書籍・Audible作品を検索する。
6. **恒久エピソードページ生成**
   - 原稿、出典、素材クレジット、関連書籍をコピーできる公開URLを生成する。
   - URLは変更しない。誤りが判明した場合は本文を訂正し、旧版を不変スナップショットとして残す。
7. **配信**
   - YouTubeへスライド動画を配信する。
   - Podcast RSSへ音声版を追加する。
   - RSSをAmazon Music for Podcastersへ登録し、Amazon Music／Audibleでの配信対象にする。
   - 各配信先の概要欄には恒久エピソードページへのURLを掲載する。

全工程完了後、生成物、出典、権利記録、処理ログをGoogle Driveへ日次バックアップする。

### 6.1 処理状態とゲート

各エピソードは次の状態機械で管理し、ゲートを飛び越えない。

```text
collected
  -> rights_passed
  -> topic_selected
  -> facts_verified
  -> script_generated
  -> script_verified
  -> media_generated
  -> publish_ready
  -> approved
  -> published
```

- ゲート失敗は `rejected`、外部障害は `blocked` とする。
- `blocked` を許可扱いして後続工程へ進めない。
- MVPでは `approved` への遷移に必ず人手操作を要求する。
- 各遷移に、入力ハッシュ、ルール版、モデル版、実行時刻、判定理由を記録する。

### 6.2 出典の独立性

「2資料以上」はURL数ではなく、情報の系統数で数える。

- Wikipediaと、そのWikipediaを転載したサイトは1系統とする。
- Wikipediaと、Wikipedia脚注にある原資料を併用する場合、Wikipedia自体を追加の独立系統として数えない。
- 複数記事が同じ一次資料だけを根拠にしている場合は1系統とする。
- 同一機関の同一データベース内にある重複レコードは1系統とする。
- 一次資料1件と、それを独自検討した信頼できる二次資料1件の組合せを優先する。
- 単一の一次資料でしか確認できない事実は、中心的主張にせず「当該資料にはこう記される」と資料帰属を明示する。

## 6A. 機械選出仕様

### 6A.1 入力特徴量

LLMを使わず、次の特徴量から候補点を計算する。

| 特徴 | 例 | 点数への影響 |
|---|---|---:|
| 月日一致 | 今日起きた出来事、人物の誕生日・命日 | 高 |
| 季節一致 | 祭り、気候、農事、年中行事 | 中 |
| ニュース語一致 | 「選挙」「鉄道」「彗星」等 | 中 |
| 固有名詞一致 | 国、都市、人物、組織、技術 | 中 |
| 資料充実度 | 独立資料数、利用可能画像数 | 高 |
| 既出類似度 | 過去回とのタイトル・固有名詞重複 | 減点 |
| 直近使用 | 同じ国・時代・人物の連続 | 減点 |

候補点の初期式を以下とする。

```text
候補点 = 日付一致×30 + 季節一致×10 + ニュース語一致×15
       + 固有名詞一致×10 + 資料充実度×25
       - 既出類似度×30 - 直近カテゴリ重複×15
```

各特徴量は0〜1へ正規化する。重みは管理画面から変更可能にする。

### 6A.2 ニュース連動の安全フィルター

ニュース本文を歴史情報の出典には使用しない。ニュースは題材選出用の単語取得だけに使用する。

以下に該当するニュースからは単語を一切採用しない。

- 死亡、殺人、傷害、事故、災害、行方不明
- 戦争、テロ、侵攻、空爆、人質
- 性犯罪、虐待、差別、ヘイト
- 疫病、重病、自殺、自傷
- 葬儀、追悼、被害者名
- 不祥事、逮捕、告発、炎上
- 当日速報、緊急速報

判定はLLMではなく、次を組み合わせる。

1. ニュース提供元のカテゴリ
2. 禁止語・類義語辞書
3. 固有表現辞書
4. URL・タグ・構造化データ
5. 見出し・カテゴリ・禁止語の定期的な再評価

禁止語が1件でも含まれるニュースは関連付け対象から除外する。判定に迷うニュースは採用しない。個人名は原則としてニュース連動語に使わず、国・地域・一般名詞・技術名だけを採用する。

速報後に事件性や被害が判明する可能性はあるが、一律の72時間待機は話題性を大きく損なうため採用しない。代わりに、初期対象カテゴリを安全な分野へ限定し、公開直前にも見出し・カテゴリ・禁止語を再取得して再判定する。公開直前の再判定に失敗した場合、その題材は当日公開しない。

安全性を優先する初期設定では、ニュース連動対象を「科学・宇宙・考古学・文化財・技術・交通・祝祭」に限定する（§5.12の安全ソースを優先使用する）。

MVPでは一般ニュースRSSを使用せず、§5.12に登録された官公庁・大学・研究機関の公式発表だけを対象とする。「交通」は事故・運休・災害情報を除き、新路線、保存車両、技術史、記念日等の許可イベント型だけに限定する。禁止語に該当しないことだけを安全判定の根拠にせず、許可カテゴリと許可イベント型の両方に一致することを必要とする。

ニュース連動を外部公開するときは、概要欄に現在のニュースとの因果関係を示唆しない。「今日の○○という話題から、関連する歴史を紹介する」程度に留め、事件・人物・組織を歴史上の否定的出来事と機械的に対応付けない。

## 7. クローラー仕様

### 7.1 取得方式

優先順位は API → RSS → 提供データセット → HTMLクロールとする。HTMLクロールは他の正式な取得方法がない場合に限定する。

### 7.2 必須保存項目

- `source_id`
- 原典URLと正規化URL
- 版・リビジョン識別子（Wikipediaのoldid、アーカイブのレファレンスコード、IIIFマニフェストID等）
- ページタイトル
- 著者・作成者・提供機関
- 公開日・資料作成日
- 取得日時
- 取得本文または根拠抜粋
- 根拠位置（ページ番号、段落ID、文字オフセット、IIIF Canvas等）
- 言語
- ライセンス名・ライセンスURL・`normalized_license_id`・利用区分（A〜D）
- 権利表示の原文
- 権利表示ページのURLとスクリーンショット
- 原資料への恒久リンク
- Wayback Machine等の外部アーカイブURL（保存申請できた場合）
- コンテンツハッシュ
- robots.txt・利用規約確認結果
- 取得経路（API、RSS、ダンプ、IIIF、HTML）とHTTP応答情報
- 保存期限と公開可否

原文全文を保存するのは、明示ライセンス、ダンプ利用条件、または内部分析上の保存根拠を確認できた場合に限る。それ以外は、必要最小限の根拠抜粋、ハッシュ、位置情報、メタデータだけを保存する。内部保存が許されても公開再配布できるとは限らないため、`storage_permission` と `publication_permission` を分離する。

### 7.3 クロール制御

- ドメインごとの同時接続数：1
- 標準待機時間：2秒以上
- `Retry-After`を遵守
- User-Agentにツール名と連絡先ページを記載
- ログイン、CAPTCHA、アクセス制御を回避しない
- robots.txtで禁止されたパスを取得しない
- 同一内容はハッシュで再取得を抑制する
- XML外部実体を無効化し、圧縮爆弾、巨大画像、偽装MIME、過大レスポンスを拒否する
- リダイレクト後も許可ドメイン内であることを検証し、プライベートIP・localhostへの接続を拒否する
- APIキー、Cookie、認証付きURL、個人情報を原文ログやLLM入力へ含めない

## 8. LLM処理

### 8.1 モデル構成

- 事実候補抽出：本番許可レジストリで指定した固定無料モデル
- 面白さ評価：本番許可レジストリで指定した固定無料モデル
- 日本語台本：NemotronまたはQwen系の、試験合格済み固定モデル
- 軽量な分類・整形：試験合格済みの小型無料モデル

無料モデルは入れ替わるため、毎日モデルAPIを確認する。`pricing.prompt == 0`、`pricing.completion == 0`、有効期限内であることに加え、JSON Schema、必要コンテキスト長、日本語回帰テスト、プロバイダーのデータ方針を検証する。

`openrouter/free` のように実モデルがランダムに変わるルーターは、本番では使用しない。フォールバックもモデルIDとプロバイダーを固定した許可リスト順とし、すべて利用不能ならジョブを `blocked` にする。モデル変更時は過去の検証用20題で回帰テストを行い、合格するまで自動公開に使用しない。

各実行で以下を保存する。

- OpenRouter上のモデルID、実プロバイダー、エンドポイント
- モデル有効期限、価格、コンテキスト長
- プロンプトテンプレート版、JSON Schema版
- 入出力ハッシュ、入力資料ID、生成パラメータ
- プロバイダーの保存・学習利用・ZDR方針
- 実トークン数と請求額

OpenRouterの設定で、学習利用を許可するプロバイダーへのルーティングを無効化する。ゼロデータ保持を必須にするかは無料エンドポイントの提供状況と両立させて設定するが、少なくとも送信先と保持方針を記録できないエンドポイントは使用しない。

### 8.2 要約出力

LLMには取得済み本文とメタデータだけを渡し、次のJSONを返させる。

```json
{
  "summary_ja": "日本語要約",
  "facts": [
    {
      "claim": "確認可能な事実",
      "evidence_quote": "根拠となる短い抜粋",
      "source_id": "source-id",
      "locator": {
        "page": null,
        "paragraph_id": null,
        "start_offset": 0,
        "end_offset": 0
      }
    }
  ],
  "people": [],
  "places": [],
  "dates": [],
  "uncertainties": []
}
```

URLとライセンスはLLMに生成・変更させず、プログラム側の値を使用する。

LLMが出力する自己申告の `confidence` は採用可否に使用しない。根拠抜粋が保存本文の指定位置と完全一致することをプログラムで検査し、主張の信頼度は「独立系統数」「資料種別」「同時代性」「明示的な不確実性」から機械計算する。

### 8.2A 主張台帳

台本生成前に、公開可能な事実だけを `claim_ledger.json` として確定する。台本モデルはこの台帳にない外部事実を追加してはならない。

```json
{
  "claim_id": "claim-001",
  "text": "公開可能な事実文",
  "evidence_ids": ["evidence-001", "evidence-002"],
  "source_family_ids": ["family-a", "family-b"],
  "reliability_score": 0.0,
  "allowed_in_script": true,
  "qualification": "断定／資料帰属／伝承／推定"
}
```

台本完成後、各文を `claim_id`、演出文、意見、接続表現のいずれかに分類する。外部検証可能な文なのに `claim_id` がない場合は公開検査を失敗させる。

### 8.3 面白さの採点

各項目を0〜10点で採点する。

| 項目 | 重み | 内容 |
|---|---:|---|
| 意外性 | 25% | 一般的な予想とのずれ |
| 物語性 | 20% | 導入・転換・結末を構成できるか |
| 信頼性 | 20% | 一次資料や複数資料で確認可能か |
| 身近さ | 10% | 現代の生活との接点 |
| 映像化適性 | 10% | 利用可能な画像・地図・年表があるか |
| 新規性 | 10% | 過去の公開内容と重複しないか |
| 安全性 | 5% | 誤解・名誉・差別等のリスクが低いか |

総合点は加重平均とする。ただし、信頼性6未満、安全性7未満、独立した根拠系統2件未満の候補は総合点に関係なく不採用とする。資料帰属を明示する単一資料紹介回は、別カテゴリとして人手承認を必須とする。

### 8.4 採点の安定化

- 同一候補を順序を変えて2回採点する。
- 2回の総合点差が2点を超えた場合は再評価する。
- 単なる有名さではなく「説明後に驚きが残るか」を評価させる。
- LLM点だけでなく、資料数、画像数、既出類似度を機械的に加味する。

## 9. 台本仕様

### 9.1 構成

1. 15秒以内の引き
2. 時代・場所の説明
3. 出来事の展開
4. 意外な点または通説との違い
5. 現代との接点
6. 不確実な点の明示
7. 出典案内

### 9.2 文章規則

- 原文の長い引用や言い換え転載を避け、複数資料から独自構成する。
- 断定の強さを資料の確度に合わせる。
- 伝説、後世の記録、同時代史料を区別する。
- 出典にない心理や会話を創作しない。
- ずんだもんの語尾は過剰にせず、内容の信頼性を損なわない。
- 読み上げ困難な固有名詞には読み仮名を付ける。
- 動画概要欄・Podcast説明欄・恒久ページに「VOICEVOX:ずんだもん」のクレジットを表示する。説明欄を表示できない配信先に備え、音声末尾にも短いクレジットを挿入できるようにする。
- VOICEVOXソフトウェア規約、ずんだもん音源利用規約、キャラクター利用ガイドラインを別々に管理する。音声利用とキャラクター画像・立ち絵の商品的利用を同一許諾とみなさない。
- MVPではずんだもんの音声だけを使用し、公式キャラクター画像・立ち絵をスライドやサムネイルへ自動使用しない。キャラクター画像を追加する場合は、収益化形態を含めて別途許諾判定する。

## 10. スライド動画仕様

- 解像度：1920×1080
- フレームレート：30fps
- 画像切替：8〜20秒を目安
- 基本要素：題名、年代、地図、人物・資料画像、短い要点、出典番号
- 画面上の文章は1スライド60文字以内を目安とする。
- 単純な画像貼り付けだけでなく、地図上の移動、年表、比較図等を自動生成する。
- 各画像に内部素材IDを持たせ、概要欄のクレジットへ対応させる。
- 素材不足時は無理に画像検索せず、著作権の発生しない自作図形・地図・年表を使用する（地図基盤はNatural Earth・地理院タイル等の許可済みデータに限る）。

## 10A. 関連書籍検索仕様

関連書籍の選出にはLLMを使用しない。

### 検索キー

- 台本で使用した人物・事件・地名・時代の正規化名称
- Wikipedia/Wikidataの識別子
- 国立国会図書館件名標目
- ISBN
- 著者名
- 出版年

### 検索先

- 国立国会図書館サーチ等の書誌API
- Open Library、Google Books等の書誌データ
- Amazon Product Advertising API（利用資格を得た場合）

Amazon/Audibleの一般検索結果をクローラーで取得しない。Amazon商品リンクの自動生成は、Amazon Product Advertising API等の正式手段を利用でき、アソシエイト規約・表示要件・価格情報の更新条件を満たす場合だけ有効化する。それまではNDL、出版社、Google Books、図書館等の非アフィリエイト書誌リンクを表示する。

### 機械ランキング

```text
関連度 = 題名一致×35 + 件名一致×25 + 人物一致×20
       + 時代一致×10 + 出版年の新しさ×5 + 音声版あり×5
```

- 題名だけの曖昧一致は採用しない。
- ISBNまたはAmazon商品識別子を確認できない商品へアフィリエイトリンクを作らない。
- 存在確認できない商品を表示しない。
- 書名・著者・ISBNが2つ以上の書誌系統で一致しない候補は表示しない。
- 関連度が閾値未満の場合は、無関係な商品を埋めるために表示せず「関連書籍なし」とする。
- 広告・アフィリエイトであることを恒久ページ上に明記する。
- Audible作品が見つからない場合は、紙・Kindle・図書館の書誌リンクだけを表示する。

## 10B. 恒久エピソードページ仕様

### URL

```text
https://<固定ドメイン>/episodes/<公開日>-<不変ID>/
```

題名変更によってURLを変えない。不変IDは公開前に確定し、再利用しない。

### 表示内容

- エピソードタイトル、公開日、更新日
- 再生プレイヤーとYouTubeリンク
- コピー可能な原稿全文
- 原稿のMarkdownダウンロード
- 主張ごとの出典番号
- 出典名、著者、提供機関、原典URL（可能な限り版指定の恒久リンク）、閲覧日
- 画像・資料・音声素材ごとのライセンスとクレジット
- 関連書籍・Audible作品
- アフィリエイト表示
- 訂正・更新履歴

### 永続性

- ページは静的HTMLとして生成する。
- 公開済みページの削除・URL変更を原則禁止する。
- 正本データをGit管理し、公開履歴を保持する。
- Google Driveとローカル/NASへ日次バックアップする。
- サービス移転時も同じURLを維持できるよう、独自ドメインの利用を推奨する。
- 外部サイト消失に備え、許諾範囲内の根拠抜粋、資料識別子、取得日時を内部保存する。
- 公開時に、規約上問題のない公開出典だけを外部アーカイブサービスへ任意保存申請し、取得できた場合は補助情報として併記する。
- 誤りが判明した場合、同じ恒久URL上の本文を最新版へ訂正し、ページ上部に訂正日・訂正内容を表示する。
- 各公開版を `/episodes/<不変ID>/versions/<version>/` に不変保存し、最新版から参照できるようにする。
- 法的削除要請や重大な権利問題では本文・メディアを非公開化できるが、可能な範囲でURLに理由と履歴を残す。

無料ホスティングだけでは永久存続を保証できないため、「独自ドメイン＋複数バックアップ」を永続URL要件とする。ドメイン維持費はLLM API費とは別扱いとする。

## 10C. Cloudflare公開基盤

### 構成

| Cloudflareサービス | 役割 |
|---|---|
| Registrar | 希望TLDが対応している場合の独自ドメイン取得・自動更新。未対応なら外部レジストラを使用 |
| DNS | サイト、音声、API用サブドメインの管理 |
| Pages | エピソードページ、出典ページ、トップページ、RSSの公開 |
| R2 | MP3、サムネイル、出典画面用画像の保存 |
| Workers | 管理API、公開前検査、RSS応答、リダイレクト |
| D1 | エピソード状態、公開履歴、訂正履歴、ジョブ状態 |

### ドメイン例

```text
https://example.jp/                         トップページ
https://example.jp/episodes/<不変ID>/      恒久エピソードページ
https://example.jp/feed.xml                 Podcast RSS
https://media.example.jp/audio/<不変ID>.mp3 R2上の音声
https://api.example.jp/                     管理API
```

### 配置方針

- HTML、CSS、JavaScript、RSSはPagesへ置く。
- MP3や大きな画像はPagesへ置かず、R2へ置く。
- R2の開発用URLではなく、`media.example.jp`等のカスタムドメインを使用する。
- MP4は原則YouTubeだけへ置き、R2へ恒久保存しない。元動画はNASとGoogle Driveへ保存する。
- 公開ページは静的生成し、D1障害時も閲覧できるようにする。
- D1は管理用メタデータに限定し、原稿・出典の唯一の保存先にしない。

### 正本とバックアップの責務

| データ | 正本 | 公開コピー | 復旧用バックアップ |
|---|---|---|---|
| コード・設定・許可レジストリ | GitHub | Pages/Workers | Google Drive・NAS |
| 公開原稿・出典・クレジット・書誌 | GitHub | Pages | Google Drive・NAS |
| Podcast MP3・公開画像 | R2 | R2カスタムドメイン | Google Drive・NAS |
| MP4 | NASまたはGoogle Drive | YouTube | 相互複製 |
| 取得原文・規約証拠・権利判定ログ | 非公開バックアップ領域 | 公開しない | Google Drive・NASの二重化 |
| 実行中ジョブ・公開状態 | D1 | 管理画面 | 日次エクスポート |
| ローカルPC | 作業中ファイルのみ | なし | 公開・バックアップ確認後に削除可 |

GitHubにはMP3、MP4、大量の原文、SQLiteファイル、生成中間物をコミットしない。公開テキストはMarkdown/JSON/YAMLで保持し、GitHubから静的サイトを再構築できるようにする。秘密鍵、APIキー、非公開原文、個人情報はGitHubへ入れない。

推奨リポジトリ構造：

```text
sourcecast/
├─ app/
├─ site/
├─ config/
│  ├─ source_registry.yaml
│  ├─ license_rules.yaml
│  └─ model_registry.yaml
├─ episodes/<year>/<episode-id>/
│  ├─ episode.yaml
│  ├─ script.md
│  ├─ claims.json
│  ├─ sources.json
│  ├─ credits.json
│  └─ books.json
└─ public/feed.xml
```

### 生成環境との分離

Cloudflareは公開・配信基盤とし、以下は自宅PC上の生成エージェントで実行する。

- クローラー
- OpenRouter呼び出し
- VOICEVOX
- スライド描画
- FFmpeg動画生成
- YouTubeアップロード

生成エージェントは完成後に、MP3・画像をR2へアップロードし、静的ページをGitリポジトリへ反映する。PagesはGit更新を検知して自動デプロイする。

### 容量方針

- 音声はMP3、64〜96kbpsを標準とする。
- 10分・64kbpsなら約4.8MB/本、毎日配信で年約1.8GBを目安とする。
- R2無料ストレージ枠を超えた場合も過去回は削除せず、超過ストレージ料金を許容する。
- サムネイルと本文画像はWebPへ変換する。
- R2バケットには削除用ライフサイクルルールを設定しない。

### 永続性・ドメイン保護

- Registrarの自動更新を有効にする。
- DNSSECを有効にする。
- Cloudflareアカウントで多要素認証を有効にする。
- ドメイン更新失敗をメールとGoogleカレンダーへ通知する。
- Pages/R2/D1の公開データを日次でNASとGoogle Driveへエクスポートする。
- Cloudflareから移転する場合も同じ独自ドメインとURLパスを維持する。

## 10D. 配信仕様

### YouTube

- 生成済みMP4、タイトル、説明、サムネイル、公開日時を登録する。
- 説明欄の先頭付近に恒久エピソードページを掲載する。
- 自動投稿開始前は非公開または限定公開でアップロードする。
- 各回で題材固有の構成、図解、根拠提示、ナレーションを生成し、テンプレート差替えだけの大量反復コンテンツにしない。
- 投稿前に、過去回との台本類似度、スライド構成類似度、サムネイル類似度を検査する。
- YouTubeの収益化・反復コンテンツ・合成コンテンツに関する規定を90日ごとに再確認し、重大な差分があれば自動投稿を停止する。

### Podcast

- MP3とRSS 2.0フィードを生成する。
- RSSにはGUID、公開日時、説明、カバー、音声URL、恒久ページURLを含める。
- GUIDは公開後に変更しない。
- RSSを標準バリデーターで検査し、MP3のContent-Type、Content-Length、Rangeリクエスト、HTTPS公開を確認する。
- 公開済み音声を差し替える場合は、訂正履歴を恒久ページとRSS説明に表示し、キャッシュ更新を考慮する。
- RSSフィードをSpotify、Apple Podcasts、Amazon Music等へ一度登録する。

### Amazon Music／Audible

- Amazon Music for PodcastersへPodcast RSSを登録する。
- Amazon側がRSSを取得する方式とし、各回をAudibleへ直接アップロードしない。
- Audible上で関連書籍を直接添付できることを前提にしない。
- RSSの説明欄に恒久エピソードページを掲載し、そのページで関連書籍・Audible作品を案内する。
- 配信先が説明欄のリンクを削除・無効化する場合に備え、音声内でも短くサイト名を案内できる設定を持つ。

## 10E. BGM・効果音仕様

- BGM・効果音は任意とする。使用する場合は画像と同じく `assets` として登録し、ライセンス・クレジット・取得記録を管理する。
- BGMはナレーションを妨げない音量とし、ラウドネス正規化とダッキング（ナレーション区間の自動減音）を行う。
- 効果音は場面転換用の数点をプリセット化し、権利判定済みのプリセットだけを毎回使い回す（回ごとの再判定を不要にする）。
- 歴史音源（SP盤等）を使う場合は、§5A第7項の3層判定（楽曲・実演・原盤）のログを必須とする。判定できない音源は使用しない。
- クレジットは概要欄と恒久ページに「曲名／作者／提供元／ライセンス」の形式で表示する。区分Cソースはサイト規定の表記方法を優先する。
- MVPではBGMなし、または「区分Cのサイト1件＋CC BYの数曲」に限定した構成とする。

## 11. 自動検査と公開条件

以下をすべて満たした場合だけ `publish_ready` とする。MVPでは、これに加えて人手承認がなければ公開しない。

- 台本中のすべての外部検証可能な事実文が主張台帳へ紐付いている。
- 中心となる主張が独立した2系統以上の資料で確認されている。
- 単一資料に帰属する記述は、台本上でも資料帰属が明示されている。
- 使用ソースの `status` が `approved` であり、規約再確認期限内である。
- 全画像・全音声素材のライセンスが許可リスト内である。
- 必要な著者名、作品名、ライセンス、原典URLが揃っている。
- 引用量が設定上限以内である。
- 転載検知：台本と各出典原文のあいだで、連続する長い一致（初期値：和文25文字以上、欧文8語以上）が検出されない。引用として明示・出所表示した箇所は除く。
- 「VOICEVOX:ずんだもん」および使用BGM・効果音のクレジットが動画概要欄と恒久ページに含まれている。
- 条件付き素材（`noc-us`、Flickr Commons、歴史音源等）に§5Aの判定ログが添付されている。
- 年代計算だけで許可された素材が、専門家レビュー前の自動公開へ混入していない。
- OpenRouterの実モデル、実プロバイダー、データ方針が本番許可レジストリと一致している。
- 音声の無音、読み飛ばし、異常な長さがない。
- 前回までの動画との類似度が規定値未満である。
- 禁止題材や危険語の検査を通過している。
- 最終成果物の出典一覧と内部記録が一致している。
- RSS、恒久ページ、YouTube説明欄の不変ID、タイトル、出典ページURLが一致している。

検査に失敗した回は公開せず、理由を記録して次候補へ進む。無理に毎日1本を成立させない。

## 13. 保存データ

主要エンティティは以下とする。

- `sources`：取得元（利用区分、`normalized_license_id`、再確認周期を含む）
- `source_registry`：候補・許可・停止・拒否の本番レジストリ
- `documents`：取得資料
- `rights_records`：権利・ライセンス証拠（正規化ID、利用区分、§5A判定結果、規約スナップショットID、権利表示スクリーンショット）
- `terms_snapshots`：ソースごとの利用規約全文の版管理
- `facts`：抽出事実と根拠
- `source_families`：出典の依存関係と独立系統
- `claim_ledger`：公開可能な主張、根拠、限定表現
- `topics`：題材候補と採点
- `news_signals`：題材選出に使ったニュース語と安全判定
- `scripts`：台本と改訂履歴
- `assets`：画像・地図・音声（BGM・効果音を含む）
- `episodes`：動画単位の情報
- `episode_pages`：不変URL、公開版、訂正履歴
- `related_books`：ISBN、書誌情報、商品リンク、関連度
- `credits`：公開クレジット
- `jobs`：処理状態とエラー
- `model_runs`：モデル、プロンプト、入出力、利用量
- `model_registry`：使用許可モデル、実プロバイダー、データ方針、回帰試験結果
- `publication_versions`：公開版、訂正理由、不変スナップショット

実行中データはローカルSQLiteとD1を使用できるが、公開原稿・出典・クレジットの正本はGitHub上のテキストファイルとする。原資料・生成物・規約証拠・日次ログはGoogle DriveとNASへ1日1回バックアップする。SQLite/D1だけを唯一の保存先にしない。

## 14. 障害時動作

- OpenRouter 429：指数バックオフ後、本番許可レジストリ内の別の固定無料モデルへ切替。無ければ `blocked`
- モデル提供終了：候補モデルを検出しても自動採用せず、回帰テストと承認完了まで `blocked`
- JSON不正：同一モデルで1回再生成後、別モデルへ切替
- クロール失敗：当日の公開を中止せず、既存候補から選定
- 権利判定失敗：該当資料・素材を即時除外
- 規約差分検知：該当ソースを一時停止し、人手確認まで新規取得を止める
- Google Drive失敗：ローカル保持し、次回再試行
- 動画生成失敗：中間成果物を保持し、工程単位で再実行
- GitHub push／Pagesデプロイ失敗：R2やYouTubeだけを先行公開せず、公開単位全体を `blocked`
- R2アップロード失敗：ハッシュ一致を確認できるまでRSSを更新しない
- YouTubeまたはPodcastの一方だけ公開された場合：恒久ページに配信状態を表示し、冪等キーで不足先だけ再試行
- バックアップ失敗：ローカル作業ファイルを削除しない

## 15. 非機能要件

- 同一入力から生成履歴を再現できる。
- APIキーをクライアントへ配布しない。
- 外部取得データを信頼しない。命令文ではなく引用データとして区切り、ツール実行権限を持たない処理へ渡し、JSON Schema検証と出力サニタイズを行う。プロンプトインジェクションを完全に無効化できるとは仮定しない。
- ダウンロードファイルの種類、容量、解像度を制限する。
- 全処理を冪等にし、再実行で重複公開しない。
- 1日のOpenRouter無料枠を超えない利用上限を設定する。
- 公開停止スイッチを管理画面の最上部に置く。
- OpenRouterのAPIキーには月額・日額の支出上限を設定し、無料モデルから有料モデルへ暗黙にフォールバックしない。
- Cloudflare、Google、GitHub、Amazonのキーを用途別に分離し、最小権限・定期ローテーション・秘密情報スキャンを適用する。
- 生成エージェントから公開基盤への操作は、対象プレフィックスと必要APIだけに限定する。
- すべての公開・訂正・削除・権利判定変更を追記型監査ログへ記録する。
- エピソードIDを全配信先の冪等キーとして使用し、二重投稿を防ぐ。
- 日次バックアップについて、月1回の復元試験を行う。バックアップ成功通知だけを復旧可能性の証拠にしない。

## 17. 受入条件

自動公開の解禁は段階制とする。

### 段階0：ユニット・固定データ試験

- 各許可ソースについて、許可例・拒否例・権利不明例を自動判定できる。
- 主張台帳にない事実文、存在しないURL、欠落クレジットを意図的に混入し、すべて公開拒否できる。
- 同一エピソードを3回実行しても、R2、GitHub、YouTube下書きが重複しない。

### 段階1：30候補のドライラン

- 公開せず30候補を生成し、権利判定・出典独立性・原稿事実を人手で全件確認する。
- 重大な権利誤判定、架空出典、裏付けのない中心的主張が1件でもあれば、修正後に30候補をやり直す。

### 段階2：限定公開10本

- YouTube限定公開と非公開Podcastフィードで10本を生成する。
- 人手確認後の修正量、音声、スライド、リンク、RSS、クレジットを測定する。
- 1本あたりの確認時間は初期目標15分以内とし、ルール改善後に5分以内を目指す。

### 段階3：公開運用30日

- 各回を人手承認して30日運用する。
- 訂正率、公開失敗、規約変更、モデル変更、配信先エラーを記録する。
- 完全自動公開は別途承認事項とし、本仕様v0.4では解禁しない。

全段階を通して以下を満たすこと。

- 権利不明素材の公開候補混入：0件
- 存在しない出典URL：0件
- 中心的主張の根拠不足：0件
- 主張台帳にない外部検証可能な事実文：0件
- 出典原文との規定超過一致（転載検知）の見逃し：0件
- クレジット表記（VOICEVOX・画像・音声素材）の欠落：0件
- 同一題材の重複生成：0件
- 禁止カテゴリのニュースとの関連付け：0件
- 公開済みエピソードURLの変更：0件
- 生成工程の手動操作なし完了率：90%以上（公開承認操作を除く）
- 段階3終了時の1本あたり手動確認時間：中央値5分以内、最大15分以内
- LLM API料金：0円
- 失敗時に誤公開せず、原因が画面で確認できる
- バックアップから任意の1回を復元し、同じ恒久ページとRSS項目を再構築できる
